# docs-idryer-org

[Русский](README.md) · **English**

The build engine that assembles the user-documentation site **docs.idryer.org** from
several product repositories.

Core principle: **a repository ≠ a site section**. The repo name never appears in the URL —
each repo is mounted into a section via a mount-path (`repos.yml`). Several repos can build
into one section; device versions are folders inside a single repo, not separate repos.

## How it works

```
push to the docs-publish branch of a product repo
  → GitHub webhook → /hooks/rebuild
  → rebuild.sh: sparse-clone repos from repos.yml → lay out into docs/<lang>/<mount>/
              → shared docs/img/ → root asset folders → resize + compress images
  → mkdocs build → nginx serves the static site
```

The build runs in a Docker container that receives the webhook. The normal workflow **needs
no SSH**: push to `docs-publish` and the site rebuilds itself.

## Connect a product repository

1. Bring the **`docs/`** in the product repo to the contract (see below).
2. Create the **`docs-publish` publication branch** — the engine reads only this branch:
   ```
   git checkout -b docs-publish && git push -u origin docs-publish
   ```
3. Register it in **`repos.yml`** (the only place that lists repos):
   ```yaml
   - mount: projects/iheater     # section path on the site (nested = group/subsection)
     url: https://github.com/pavluchenkor/iHeater.git
   ```
4. Add a menu item — `docs/ru/.pages` and `docs/en/.pages`.
5. Add a **webhook** to the repo: Settings → Webhooks → Add. Payload
   `https://<domain>/hooks/rebuild`, Content type `application/json`,
   Secret = `WEBHOOK_SECRET` (from the server), event `push`.

A private repo needs access for the server-side `GITHUB_TOKEN` (fine-grained, Contents: Read).

## Product repository contract

```
<repo>/                         # docs-publish branch
  docs/
    ru/  en/                    # folder-based i18n (ru/en are sources; others auto-translated)
      .pages                    # section title + order (awesome-pages)
      01-<section>/…            # numeric prefixes define the order
      README.md                 # section cover
    img/<section>/…             # images, SHARED across all languages (links ../../img/…)
  CAD/ KiCad/ PDF/ Schematics/  # binaries next to docs/ → /assets/<key>/… (via hook)
```

- `docs/ru` is required (the primary language). Folder names must match across languages.
- Images live in one shared `docs/img/`, not duplicated per language.
- Binary files (CAD/PDF/…) sit in the repo root; a link like `../../PDF/x.pdf` works both on
  GitHub and on the site — `hooks/asset_links.py` rewrites it to `/assets/<key>/PDF/x.pdf`.

## Checks before publishing

Two scripts validate a product repo's `docs/` locally (no key/network needed):

```
python3 validate-i18n.py <repo>/docs   # translation gaps: which .md are missing per language (ru reference) + extra files
python3 check-links.py  <repo>/docs    # broken links to images/files in .md, case-sensitive (the Linux server is case-sensitive)
```

## Languages and auto-translation

Sources are `ru` and `en`. Other languages are generated by `translate-docs.py` (auto-translates
`.md` from `docs/ru`). Language codes use hyphens (BCP47): `pt`, `pt-BR`, `zh`, `zh-Hant` (not
`pt_BR`, not `zh_CN`). The language set must match in `rebuild.sh` (`LANGUAGES`) and `mkdocs.yml`
(`plugins.i18n`). The translator needs an external translation API key (an environment variable —
see the script itself). It translates changed `ru` files and **fills in missing ones** per
language; `--dry-run` prints the plan and an API-request estimate without translating (no key needed).

## What requires action on the server

`rebuild.sh` re-clones the engine on every build, so changes to **content/config** (`mkdocs.yml`,
`repos.yml`, `docs/`, `hooks/*.py`) are picked up automatically. But **mounted** files
(`rebuild.sh`, `hooks.json.template`) and the `Dockerfile` require a `git pull` on the server
(and `docker compose build` for the `Dockerfile`).
