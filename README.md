# docs-idryer-org

**Русский** · [English](README.en.md)

Движок, который собирает сайт пользовательской документации **docs.idryer.org** из
нескольких продуктовых репозиториев.

Ключевой принцип: **репозиторий ≠ раздел сайта**. Имя репо не попадает в URL — каждый
репо монтируется в раздел через mount-path (`repos.yml`). Несколько репо могут собираться
в один раздел; версии устройства — это папки внутри одного репо, а не разные репо.

## Как работает

```
push в ветку docs-publish продуктового репо
  → GitHub webhook → /hooks/rebuild
  → rebuild.sh: sparse-clone репо из repos.yml → раскладка в docs/<lang>/<mount>/
              → общий docs/img/ → корневые asset-папки → ресайз + сжатие картинок
  → mkdocs build → nginx отдаёт статику
```

Сборка идёт в Docker-контейнере, который принимает webhook. Обычный сценарий **не требует
SSH**: пушнул в `docs-publish` — сайт пересобрался сам.

## Подключить продуктовый репозиторий

1. Привести **`docs/`** в продуктовом репо к контракту (см. ниже).
2. Завести **ветку публикации `docs-publish`** — движок читает только её:
   ```
   git checkout -b docs-publish && git push -u origin docs-publish
   ```
3. Прописать в **`repos.yml`** (единственное место со списком репо):
   ```yaml
   - mount: projects/iheater     # путь раздела на сайте (вложенный = группа/подраздел)
     url: https://github.com/pavluchenkor/iHeater.git
   ```
4. Добавить пункт в меню — `docs/ru/.pages` и `docs/en/.pages`.
5. Повесить **webhook** на репо: Settings → Webhooks → Add. Payload
   `https://<домен>/hooks/rebuild`, Content type `application/json`,
   Secret = `WEBHOOK_SECRET` (с сервера), событие `push`.

Приватному репо нужен доступ для серверного `GITHUB_TOKEN` (fine-grained, Contents: Read).

## Контракт продуктового репозитория

```
<repo>/                         # ветка docs-publish
  docs/
    ru/  en/                    # папочный i18n (ru/en — исходники; прочие — автоперевод)
      .pages                    # заголовок раздела + порядок (awesome-pages)
      01-<секция>/…             # числовые префиксы задают порядок
      README.md                 # обложка раздела
    img/<секция>/…              # картинки, ОБЩИЕ на все языки (ссылки ../../img/…)
  CAD/ KiCad/ PDF/ Schematics/  # бинарь рядом с docs/ → /assets/<key>/… (через хук)
```

- `docs/ru` обязателен (основной язык). Имена папок должны совпадать между языками.
- Картинки — одна общая `docs/img/`, не дублируются по языкам.
- Бинарные файлы (CAD/PDF/…) лежат в корне репо; ссылка вида `../../PDF/x.pdf` работает и
  в GitHub, и на сайте — `hooks/asset_links.py` переписывает её в `/assets/<key>/PDF/x.pdf`.

## Проверка перед публикацией

Два скрипта проверяют `docs/` продуктового репо локально (ключ/сеть не нужны):

```
python3 validate-i18n.py <repo>/docs   # пробелы перевода: каких .md нет в языках (эталон ru) + лишние файлы
python3 check-links.py  <repo>/docs    # битые ссылки на картинки/файлы в .md, с учётом регистра (сервер Linux регистрозависим)
```

## Языки и автоперевод

Исходники — `ru` и `en`. Остальные языки генерирует `translate-docs.py` (автоперевод `.md`
из `docs/ru`). Коды языков — через дефис (BCP47): `pt`, `pt-BR`, `zh`, `zh-Hant` (не `pt_BR`,
не `zh_CN`). Набор языков должен совпадать в `rebuild.sh` (`LANGUAGES`) и `mkdocs.yml`
(`plugins.i18n`). Переводчику нужен ключ внешнего API перевода (переменная окружения — см.
сам скрипт). Скрипт переводит изменённые `ru`-файлы и **добивает недостающие** в каждом
языке; `--dry-run` показывает план и оценку числа запросов к API без перевода (ключ не нужен).

## Что требует действий на сервере

`rebuild.sh` клонирует движок заново каждую сборку, поэтому правки **контента/конфига**
(`mkdocs.yml`, `repos.yml`, `docs/`, `hooks/*.py`) подхватываются сами. А **примонтированные**
файлы (`rebuild.sh`, `hooks.json.template`) и `Dockerfile` требуют `git pull` на сервере
(для `Dockerfile` ещё `docker compose build`).
