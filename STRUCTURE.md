# Структура продуктового репозитория

Референс файловой структуры, к которой надо привести продуктовый репозиторий,
чтобы движок `docs-idryer-org` собрал из него раздел сайта.

- **Готовый скелет для копирования:** [`template/`](template/) — скопируй в новый репо и переименуй.
- **Живой эталон:** `reference-repos/Build-Your-Own-iDryer/docs` (канонический образец).
- **Формальный контракт:** раздел «Контракт продуктового репозитория» в [README.md](README.md).

## Дерево

```
<repo>/                         ← ветка docs-publish
├── docs/
│   ├── ru/                     ← основной язык (ОБЯЗАТЕЛЕН)
│   │   ├── .pages              ← title: + порядок верхнего уровня (awesome-pages)
│   │   ├── README.md           ← обложка раздела (движок отдаёт как index)
│   │   └── 01-секция/          ← числовой префикс = порядок
│   │       ├── .pages          ← title: «Название секции»
│   │       ├── 01-страница.md  ← kebab-case, числовой префикс
│   │       └── 02-страница.md
│   ├── en/                     ← второй язык-исходник (структура та же, что в ru)
│   │   └── … (зеркало ru)
│   └── img/                    ← ОДНА общая папка картинок на все языки
│       └── секция/             ← группировка по секции — твоя конвенция, движку всё равно
│           └── example.jpg
├── CAD/  KiCad/  PDF/  Schematics/   ← бинарь в КОРНЕ репо → /assets/<key>/… (список фиксирован!)
└── (прочие языки)              ← НЕ вручную: их генерирует translate-docs.py из ru
```

## Правила

1. **`docs/ru` обязателен** — основной язык, fallback для остальных. `en` — второй исходник.
   Прочие языки создаёт автоперевод (`translate-docs.py`), руками не заводим.
2. **Имена папок должны совпадать между языками** (`ru` и `en`): `01-секция/` в обоих.
   Иначе `rebuild.sh` не свяжет переводы и уронит раздел в fallback.
3. **Порядок** — числовыми префиксами (`01-`, `02-`…). Имена файлов — `kebab-case`.
4. **`.pages`** в каждой папке: `title:` задаёт имя раздела/секции в меню сайта.
5. **`README.md`** в папке = обложка (index) этого раздела.
6. **Картинки — одна общая `docs/img/`**, не дублируются по языкам. Группировка
   по секциям (`docs/img/<секция>/`) — организационная конвенция (для порядка):
   движок копирует `docs/img` целиком и не интерпретирует имена подпапок — раскладывай
   как удобно, лишь бы ссылки вели на реальные файлы.
   - Пиши **естественную относительную ссылку** от своего файла:
     `![](../../img/<секция>/файл.jpg)` для страницы `docs/<lang>/<секция>/page.md`.
     Она верна в исходном дереве — **картинка видна в превью VS Code и на GitHub**.
   - Про глубину mount думать НЕ надо: хук `asset_links.py` при сборке переписывает
     `…/img/...` в абсолютный `/img/...`, который работает на сайте независимо от mount.
   - Обязательно указывай путь внутри `img` (`…/<секция>/файл.jpg`) — частая ошибка
     забыть сегмент секции.
7. **Регистр имён важен**: сервер — Linux (регистрозависимый). `Box.jpg` ≠ `box.jpg`.
   На macOS ошибка не видна — проверяй `check-links.py`.
8. **Бинарь — в КОРНЕ репо**, не внутри `docs/`. Хук `asset_links.py` превращает
   ссылки в `/assets/<key>/…`. Ссылка из страницы вида `../../PDF/x.pdf` работает и
   на GitHub, и на сайте.
   - **Список папок ФИКСИРОВАН: `CAD`, `KiCad`, `PDF`, `Schematics`.** Только их движок
     забирает из репо (sparse-checkout). Папку с другим именем он **не подхватит** —
     файлы не попадут на сайт.
   - Добавить новую asset-папку = прописать её в ДВУХ местах движка: `EXTRA_ASSET_DIRS`
     в `rebuild.sh` и `extra.extra_asset_dirs` в `mkdocs.yml` (правка `rebuild.sh`
     требует `git pull` на сервере).
9. **`.DS_Store`** и прочий мусор — в `.gitignore`, в репо не держим.

## Меню и порядок (`.pages`)

По умолчанию порядок задают числовые префиксы файлов, подписи — из заголовка `# H1`.
Через `nav:` в `.pages` порядок и подписи можно задать явно и сгруппировать страницы
в подразделы. Пути в `nav:` — **только в пределах папки этого `.pages`** (своё поддерево;
сослаться на файл из другой папки нельзя).

**Вариант 1 — свой порядок + человеческие подписи (плоско):**

```yaml
title: v3
nav:
  - Обзор: overview.md
  - Перед сборкой: before.md
  - Комплект KIT: kit.md
  - Сборка шкафа: cabinet.md
  - PIR-Box: pir-box.md
  - Детали для печати: pir-box-cad.md
```

Подпись слева от `:` переопределяет заголовок в меню; порядок — как в списке.
Имена справа — **реальные имена файлов на диске**. Раз порядок задаёт `nav:`,
числовые префиксы в именах не нужны. Но если файлы названы с префиксами
(`01-overview.md`), в `nav:` указывай их **с префиксом** — имя должно совпадать.

**Вариант 2 — вложенные подразделы (другая структура меню):**

```yaml
title: v3
nav:
  - Обзор: overview.md
  - Подготовка:
      - before.md
      - kit.md
  - Сборка:
      - cabinet.md
      - PIR-Box:
          - pir-box.md
          - pir-box-cad.md
```

Группы (`Подготовка`, `PIR-Box`) — **виртуальные заголовки меню**: они не меняют URL и
физическое расположение файлов, а значит и не влияют на число `../` в ссылках.
Имя в `nav:` должно точно совпадать с именем файла на диске (`.pages` не переименовывает).
Задал явный `nav:` — перечисли все нужные файлы или добавь `- ...` (подставит остальные).

> **Путь папки в `nav:` — БЕЗ слеша в конце.**
> `- Установка: installation` → раскрывается в **секцию** с вложенными страницами ✅
> `- Установка: installation/` (со слешем) → станет простой **ссылкой** и НЕ развернёт
> вложенное дерево ❌. Для файлов (`page.md`) слеша и так нет.

## Проверка перед публикацией

```
python3 validate-i18n.py <repo>/docs     # каких .md не хватает в языках (эталон ru) + лишние
python3 check-links.py  <repo>/docs      # битые ссылки на картинки/файлы (с учётом регистра)
python3 check-nav.py    <repo>/docs/ru   # дерево меню из .pages + предупреждения (напр. слеши)
```

## Подключение

Когда структура готова: ветка `docs-publish`, запись в `repos.yml` (`mount: …`), webhook.
Подробнее — раздел «Подключить продуктовый репозиторий» в [README.md](README.md).
